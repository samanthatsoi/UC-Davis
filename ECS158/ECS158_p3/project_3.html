3 Preliminary information

Both programs of this project assignment use the message passing paradigm in other to create cooperating tasks.

The most popular message passing interface is incidentally called MPI (Message Passing Interface), and are several implementations via software libraries. OpenMPI is one of these implementations.

3.1 Compilation

In order to compile a program using OpenMPI, the easiest is to use mpicc. It is a wrapper around GCC and will automatically link your program with the OpenMPI library without having to specify any linker argument.

  • To use the OpenMPI API in your program, include header mpi.h.
  • Modify your Makefile to use mpicc.

Reference: man mpicc

3.2 Execution

MPI programs are run via a wrapper as well, called mpirun.

$ mpirun [mpi options] <program> [program arguments]

Without any MPI options, mpirun will simply create as many copies of the program as there are processors on the local system. Otherwise it is possible via option -n (or -np) to specify another number of copies.

In order to run an MPI program on multiple computers, and have the message passing be done over the network, it is possible to specify a file containing the list of other online computers.

First, you need to setup SSH to allow password-less connections. If you do not have a public key (e.g., ~/.ssh/id_rsa.pub), then create one:

$ ssh-keygen -t rsa
...

Once you have a public key, add the contents of it to the list of authorized keys:

$ cat ~/.ssh/id_rsa.pub >> ~/.ssh/authorized_keys

You can verify that you can connect to another computer without having to retype the password:

user@pc1 $ ssh pc2
...
user@pc2 $ exit
Connection to pc2 closed.
user@pc1 $

Once you have configured password-less SSH connections, you will find a script at /home/cs158/public/p3/create_csif_hostfile.sh that can generate such an up-to-date host file for the CSIF computers.

$ /home/cs158/public/p3/create_csif_hostfile.sh
pc1: OK
pc2: OK
pc3: OK
...
$ ls -1
...
csif_hostfile
...
$ mpirun --hostfile csif_hostfile -np 4 <program> [program arguments]
...
$

It is suggested to run this script at least once every day when running your benchmarks, as it gets populated with the computers that are available only. If the hostfile contains non-reachable computers, mpirun might just hang indefinitely.

Reference: man mpirun

4 Program #1: Matrix-multiplication with message passing

4.1 Introduction

With POSIX threads and OpenMP, a single process can create many cooperating threads, each working on a separate segment of the dataset, in order to increase performance. Such approach is adapted to shared-memory system, since threads of a same process are necessarily sharing the same address space.

When using clusters of physically distinct computers, threads are not adapted (although approaches such as Distributed-Shared Memory make it possible) and cooperating tasks usually opt for a message-passing paradigm.

4.2 Goal

In this first part of the assignment, you will use MPI to parallelize the matrix multiplication program from Project 1.

Based on the best i-j-k permutation from Project 1, write a program named mmm_mpi.c that receives one command line argument, which defines the matrix order. The program should display the time it took to complete the matrix multiplication, as well as some checksum values to make sure that the result of the product is correct, and that the operands have not been changed.

4.3 Additional information and constraints

  • Right from the beginning of the program, and after each process gets their rank, the master process and the worker process should start executing different code.
  • The workers’ function should receive all the parameters that it needs from MPI messages (e.g. the order of the matrices, the segment of rows it should be working on, etc.), perform the local computation, and send back the results to the master.
    • The workers should not allocate more memory than what they need.
  • The master should start by analysing the command arguments, then broadcast the matrix order to all the workers, and then distribute their workload. Finally, it can receive the results from the workers and reconstitute the result matrix.
  • The matrix is not bounded anymore, so that you can run very big matrix multiplications and see how the increase of workers can handle the workload.

5 Program #2: Mandelbrot set

5.1 Introduction

The Mandelbrot set is a famous set of complex numbers that all share a same interesting property and produce beautiful fractal shapes.

To determine if a complex is part of the Mandelbrot set, we start with initial complex value and we repetitively iterate the equation . Complex number is part of Mandelbrot set if the result of the series stays bounded as becomes big enough.

5.2 Mapping to an image

Because the plane of complex numbers is infinite, we can only consider a finite portion of it. For this program, we consider a square area of 1024x1024 points, that we can then transform into pixels in order to generate a fractal image like above.

In order to associate these 1024x1024 points to a set of complex numbers, the program receives via command line arguments:

  • The center coordinate of the plot area x_center and y_center.
  • The zoom level z, which gives the distance between points on each axis as .

For example, if given x_center = -1.0, y_center = 0.5 and z = 8:

  • The distance between each point is .
  • The requested plot area is wide by wide, center at .

In the plot area, when considering a point at coordinate , the corresponding complex number would be .

For each point of the plot area, the series computation is run until the magnitude of the result, , becomes larger than (which means that the complex number is not bounded, and therefore does not belong to the Mandelbrot set), or until a predefined maximum number of iteration. The maximum number of iterations, or cutoff, is also received by the program as command line argument.

This number of iterations, for each point, is the result that we are representing graphically. White pixels will be associated to points belong to the Mandelbrot set, and pixels on the grayscale (up until black) are associated to points which did not stay bounded before the cutoff number of iterations.

In the pixel matrix, the pixel [0][0] corresponds to the point of the plot area.

5.3 Goal

Write program mandelbrot_mpi.c that receives four command line arguments.

  • The first argument defines the x coordinate of the plot’s area center. This parameter can be between -10.0 and 10.0.
  • The second argument defines the y coordinate of the plot’s area center. This parameter can be between -10.0 and 10.0.
  • The third argument specifies the zoom level. It can be between 0 and 100.
  • The fourth argument gives the cutoff, that is the maximum number of iterations to run the series on a complex number, to determine whether the series stays bounded or escapes to infinity. We arbitrarily decide the cutoff value should be between 50 and 1000.

The program should time the computational phase and display the running time in the console. Finally, it should run the matrix checksum on the array representing the 1024x1024 points, and save the matrix as a PGM image.

5.3.1 Additional information and constraints

  • Right from the beginning of the program, and after each process gets their rank, the master process and the worker process should start executing different code.
    • If there are less than 2 processes, the master should quit with error message Error: not enough tasks.
  • The workers’ function should receive all the parameters that it needs from MPI messages (e.g. the coordinates of the plot area, the distance between points, and the cutoff value), and then dynamically get the row index (corresponding to a specific row in the plot area) it should work on from the master. Once it has computed the Mandelbrot set for each point of the given row, the results can be sent back to the master.
  • On the other hand, the master should start by analysing the command line arguments, and then broadcast the necessary parameters to all the workers. Then it should distribute dynamically all the rows of the plot area to the workers, until the entire area has been computed.
  • Because the work repartition should be dynamic, you need to provide a way for the workers to know when there is no more rows to be processed so that they can stop their processing loop and exit the program. You can look into MPI tags for this aspect.
  • For the image generation, use the PGM format in its ASCII version.
    • The filename should start with mandel_, and then contain the arguments given on the command line. The extension should be .pgm. See example above in the execution trace.
    • The first line contains a tag (P2 in our case), the second line contains the image size (“1024 1024” in our case, for an image of 1024x1024), the third line defines the maximum value for the grayscale (should be the cutoff maximum of iterations in our case), and then each line contains the results for each computed row.
    • For the lines of results, you should not bother gracefully dealing with trailing white spaces. They will be ignored.
